# Seedance v1.1.4 统一模板任务接入方案（精简版）

发布日期：2026-05-08

## 1. 版本定位

`v1.1.4` 的目标不是重做任务系统，而是把当前三条核心链路统一接入现有任务中心：

1. `seedance_material_package_tool`
2. `happyhorse_media_tool`
3. `gpt_image2_media_tool`

一句话：**不换底座、先统一行为与数据入口**。

## 2. 当前问题（必须解决）

1. 任务页数据源是 `/api/v1/tasks`，但底层仍主要依赖 `material_packages` 任务记录。
2. HappyHorse 任务提交/轮询后没有统一落盘，所以任务页看不到。
3. GPT-Image2 使用独立本地任务目录（`.data-dev/gpt_image2_tasks`），未并入统一任务页。
4. 不同工具返回字段和状态语义不一致，前端需要做额外兼容。

## 3. 设计原则

1. **小改动优先**：不引入新数据库，不做大迁移。
2. **统一入口优先**：任务页只认 `/api/v1/tasks/*`。
3. **后端接口优先稳定**：`skill/tool/prompt` 视为可替换层，后端契约长期稳定。
4. **工具无侵入优先**：工具主要增加“上报任务状态”动作，不重写业务逻辑。
5. **下载可靠优先**：视频统一走后端归档，避免上游 URL 过期。

## 3.1 后端健壮性要求（冻结）

1. `Task Bridge` 作为唯一上报契约，模板层可重写，但必须遵守桥接字段规范。
2. 上报接口必须幂等：同一任务重复上报不会产生重复记录或破坏状态。
3. 写入失败不应阻断主生成流程：上报失败记录日志并可重试。
4. 状态流转要容错：乱序上报、重复上报、部分字段缺失都可安全处理。
5. 对未知扩展字段保持前向兼容：忽略未知字段，不影响已有流程。
6. 提供可观测性：按 provider 统计上报成功率、刷新成功率、下载成功率和错误码分布。

## 4. 精简架构

### 4.1 保持现有存储

继续使用现有任务记录目录（`material_packages/.../tasks/*.json`）作为统一任务事实来源。

### 4.2 新增 Task Bridge（核心）

新增内部桥接接口（或同等内部服务函数），让任意 template tool 能上报任务状态到统一任务记录。

建议接口：`POST /api/v1/tasks/bridge/upsert`

最小入参：

1. `task_id`
2. `provider`（`ark` / `happyhorse` / `openai_image2` ...）
3. `provider_task_id`
4. `tool_name`
5. `skill_name`
6. `status`
7. `raw_submit_response`（可选）
8. `raw_last_response`（可选）
9. `video_url` / `image_urls`（可选）
10. `chat_id`、`model`、`request_id`、`error_code`、`error_message`（可选）

桥接层只做三件事：

1. 状态归一化（`PENDING/RUNNING/SUCCEEDED/FAILED/CANCELED`）。
2. 写入/更新统一任务记录（调用现有 `_touch_task_record` 体系）。
3. 对视频成功任务触发归档轮询（复用现有归档逻辑）。

同时满足两条硬约束：

1. 对重复上报天然幂等（建议按 `provider + provider_task_id` 识别同一任务）。
2. 返回统一错误结构，便于模板层统一重试与告警。

### 4.3 Provider 刷新分发

把“任务状态刷新”从 Ark 单一路径改为按 `provider` 分发：

1. `ark`：复用现有 Ark 查询逻辑。
2. `happyhorse`：新增 DashScope `tasks/{id}` 查询适配。
3. `openai_image2`：不走远端轮询，主要依赖工具异步执行时主动上报。

## 5. 统一任务记录最小扩展字段

在现有任务 JSON 上补最小字段，不重构主模型：

1. `artifact_kind`：`video` / `image`
2. `image_urls`：图片任务产物列表（可为空）
3. `primary_image_url`：首图 URL（可用于任务卡片）

已有字段继续复用：

1. `provider/provider_task_id/tool_name/skill_name`
2. `status/archive_status/download_ready`
3. `thumbnail_url/video_preview_url/video_download_url`

## 6. 三类工具接入规范（统一行为）

### 6.1 Seedance

1. 保持现有链路。
2. 提交后通过桥接层落盘（已有路径可直接对齐）。
3. 查询状态时继续更新 `raw_last_response`，保持自动归档。

### 6.2 HappyHorse

1. `generate_*` 成功后立即上报 `PENDING`。
2. `get_*_task_status` / `wait_*` 每次查询后上报最新状态。
3. `SUCCEEDED` 且拿到 `video_url` 后触发统一归档，任务页可见并可下载。

### 6.3 GPT-Image2

1. 提交时上报 `PENDING`。
2. 后台执行线程进入 `RUNNING` 时上报。
3. 产图上传成功后上报 `SUCCEEDED + image_urls + primary_image_url`。
4. 失败上报 `FAILED + error_*`。
5. 保留原本地任务存储一段时间（双写），稳定后下线。

## 7. 前端任务页最小改造

不改入口与布局规则，继续使用：

1. 网格 `3~8` 列响应式。
2. Cell 比例 `3:4`。
3. 卡片与浮层下载图标；浮层保留删除图标（按权限）。

仅补充媒体类型分支：

1. `artifact_kind=video`：沿用现有视频预览与下载。
2. `artifact_kind=image`：卡片显示首图，浮层展示大图；下载按图片产物处理。

## 8. 下载与过期策略

1. **视频任务**：成功后自动下载到服务器归档；用户下载只走平台下载接口。
2. **图片任务**：优先使用平台内部文件存储 URL（非上游临时链接）。
3. 目标是避免“上游任务过期后不可下载”的问题。

## 9. 分阶段发布（建议）

### Phase A（1-2 天）

1. Task Bridge + 状态归一化。
2. HappyHorse 接入统一上报。

### Phase B（1-2 天）

1. GPT-Image2 双写接入统一上报。
2. 任务页图片分支渲染。

### Phase C（0.5-1 天）

1. 验证后下线 GPT-Image2 独立任务读取路径。
2. 完成文档与回归测试。

## 10. 验收标准（DoD）

1. 同一任务页可看到 Seedance、HappyHorse、GPT-Image2 任务。
2. 可按用户筛选且全员可见规则保持不变。
3. HappyHorse 成功任务可自动归档并长期下载。
4. GPT-Image2 成功任务可展示首图并下载。
5. 新增一个模板工具时，只需接入 Task Bridge，不改任务页主体逻辑。
