# Seedance v1.1.5 需求文档（分组 Key 路由）

发布日期：2026-05-08

## 1. 版本定位

`v1.1.5` 目标是把 Seedance 的 API Key 选择从模板层迁到后端，按 OpenWebUI 用户组做稳定路由。

一句话：**一个工具入口，后端按组分配 Key，费用在厂商平台看。**

## 2. 背景与问题

当前需要为不同用户分配不同 Seedance API Key，以便在厂商平台区分用量与费用。

如果把 Key 放在 `skill/tool` 层，会有三个问题：

1. 可绕过，后续换 skill/tool 时行为不稳定。
2. 维护成本高，新增 provider 时需要复制多套模板。
3. 密钥策略和业务提示词耦合，不利于治理。

## 3. 目标

1. 保持一个 Seedance 工具入口，不按 key 分叉工具。
2. 后端按用户组选择 API Key。
3. 与 OpenWebUI 分组机制打通，管理员改组成员即可生效。
4. 任务记录保留 `credential_alias`，支持排查与对账定位。
5. 在「设置 > 关于」显示二开版本信息：`v x.x.x` + `build xxxxx`。
6. 对 LLM 暴露稳定错误类型，便于按错误码给出明确下一步动作。

## 4. 非目标

1. 本版本不做本地计费统计与金额计算。
2. 本版本不改 HappyHorse / GPT-Image-2 的 key 路由，仅预留扩展位。
3. 本版本不引入新数据库。

## 5. 设计原则

1. 后端契约稳定优先，`skill/tool/prompt` 视为可替换层。
2. 默认安全优先：未配置分组映射不放行请求。
3. 配置可扩展：同一套路由结构可复用于多 provider。
4. 行为可观测：能快速定位“哪个用户走了哪个 key 别名”。

## 6. 配置模型

建议新增配置文件：`config/key_routing.json`

示例：

```json
{
  "providers": {
    "seedance": {
      "strict_single_group": true,
      "default_alias": null,
      "credentials": {
        "k1": { "env": "ARK_API_KEY_SEEDANCE_K1" },
        "k2": { "env": "ARK_API_KEY_SEEDANCE_K2" },
        "k3": { "env": "ARK_API_KEY_SEEDANCE_K3" }
      },
      "bindings": [
        { "group_id": "grp_seedance_k1", "alias": "k1", "priority": 100 },
        { "group_id": "grp_seedance_k2", "alias": "k2", "priority": 100 },
        { "group_id": "grp_seedance_k3", "alias": "k3", "priority": 100 }
      ]
    }
  }
}
```

环境变量示例：

```env
ARK_API_KEY_SEEDANCE_K1=xxx
ARK_API_KEY_SEEDANCE_K2=yyy
ARK_API_KEY_SEEDANCE_K3=zzz
```

约束：

1. `group_id` 优先于 `group_name`，避免名称变更导致路由漂移。
2. `alias` 只作为内部标识，不直接暴露明文 key。
3. `default_alias` 默认建议 `null`。

## 7. 路由决策规则

后端在每次 Seedance 提交/状态查询前执行：

1. 获取当前用户所属组（OpenWebUI Groups）。
2. 找出该 provider 下命中的绑定关系。
3. 在 `strict_single_group=true` 时：
1. 命中 0 个：拒绝，返回“未配置分组路由”。
2. 命中 1 个：放行，选中该 alias 对应 key。
3. 命中 >1 个：拒绝，返回“分组冲突”。
4. 若策略允许 fallback，再走 `default_alias`。

## 8. 后端改造范围

## 8.1 密钥解析层

新增统一方法（命名可调整）：

1. `resolve_provider_credential(provider, user_id, db)` -> `alias + api_key`
2. `resolve_user_group_ids(user_id, db)` -> `group_ids`

## 8.2 Seedance 调用点接入

替换当前固定 `ARK_API_KEY` 的读取方式，在所有 Seedance provider 请求路径统一走路由解析。

## 8.3 任务记录补充字段

在任务记录中新增并维护：

1. `credential_alias`
2. `routing_group_id`（可选）

说明：

1. 仅记录 alias，不记录明文 key。
2. 状态查询应优先复用任务提交时写入的 alias，保证同 key 查询。

## 8.4 错误契约（面向 tool/skill/LLM）

后端新增并冻结以下错误码：

1. `KEY_ROUTING_NO_GROUP`
2. `KEY_ROUTING_MULTI_GROUP`
3. `KEY_ROUTING_ALIAS_NOT_FOUND`
4. `KEY_ROUTING_ENV_MISSING`
5. `KEY_ROUTING_PROVIDER_NOT_CONFIGURED`
6. `KEY_ROUTING_RESOLVE_FAILED`

tool 返回结构约束：

1. 统一透传后端错误字段：`ok`、`status_code`、`error_code`、`error_message`、`request_id`。
2. 不在 tool 内部把路由错误改写成其他自定义文案。
3. 仅做必要格式归一化，不丢失 `error_code` 语义。

skill 侧行为约束（小改）：

1. 继续原样回传 `error_code/error_message/request_id`。
2. 按错误码输出下一步建议：
1. `KEY_ROUTING_NO_GROUP` / `KEY_ROUTING_MULTI_GROUP`：提示联系管理员检查用户组绑定。
2. `KEY_ROUTING_ALIAS_NOT_FOUND` / `KEY_ROUTING_ENV_MISSING`：提示管理员检查路由配置与环境变量。
3. `KEY_ROUTING_PROVIDER_NOT_CONFIGURED` / `KEY_ROUTING_RESOLVE_FAILED`：提示管理员检查 provider 路由总配置和服务日志。

## 9. 管理操作流程（管理员视角）

1. 在 OpenWebUI 创建组（如 `seedance-k1/k2/k3`）。
2. 把用户加入对应组。
3. 在 `key_routing.json` 维护 `group_id -> alias`。
4. 在 env 设置 alias 对应 key。
5. 重启服务或加载配置。

## 10. 测试方案（最小可行）

1. 准备 3 组 + 3 用户，每人仅属于 1 个组。
2. 每个用户提交 1 个 Seedance 任务。
3. 校验任务记录的 `credential_alias` 是否分别命中 `k1/k2/k3`。
4. 在厂商平台确认对应 key 用量变化。
5. 反例 A：无组用户提交，预期拒绝。
6. 反例 B：同用户属于两个 seedance 组，预期拒绝。
7. 校验 tool 返回包含 `error_code` 且为约定值。
8. 校验 skill 对不同错误码给出正确动作建议。

## 11. 验收标准（DoD）

1. 分组变更后无需改 skill/tool 即可切换 key 路由。
2. 同一用户在相同配置下连续提交任务，`credential_alias` 稳定一致。
3. 无组和多组冲突场景均被正确阻断并返回可理解错误。
4. 任务链路不回归：提交、刷新、归档、下载保持可用。
5. 不引入 prod 破坏性变更（先 dev 验证）。
6. 在「设置 > 关于」可见二开版本展示，格式为 `v x.x.x` 和 `build xxxxx`。
7. 发生路由异常时，LLM 能看到稳定错误码并给出正确处理建议。

## 12. 附加需求：关于页二开版本展示

展示位置：

1. 前端「设置 > 关于」页面新增一行（或两行）二开版本信息。

展示格式：

1. `v x.x.x`：业务/发布大版本。
2. `build xxxxx`：二开开发构建版本，改为构建时间戳。

build 字段规范：

1. 推荐格式：`YYYYMMDD_HHmm`（示例：`20260508_1542`）。
2. 时间基准：服务部署时区（当前建议 `Asia/Shanghai`）。
3. 建议可选附带短提交号用于排障（示例：`20260508_1542+3fa922fcf`）。

建议来源：

1. 后端配置或环境变量统一下发，前端只读展示。
2. 建议预留字段：`fork_version`、`fork_build`。
3. 未配置时显示 `unknown`，避免页面空值。
