# 21-v1.1 项目总体说明

## 1. 文档目的
本文用于统一说明当前 `open-webui-seedance-core` 的改造范围、实现结构与运行方式，便于开发、联调、部署和后续迭代。

当前项目可分为两大部分：

1. 界面改造：在 OpenWebUI 聊天输入区增加“上传素材/引用素材”入口，承接素材工作流。
2. 媒体处理流程：完成素材解析、存储、引用校验、生成提交、任务查询，并配套 Tool / Skill / System Prompt。

---

## 2. 总体架构

### 2.1 分层
1. 前端交互层（Svelte）  
负责素材上传入口、素材浏览与引用插入。

2. 后端 API 层（FastAPI）  
负责素材落盘/入库（TOS）、引用解析、生成任务提交与状态查询。

3. 编排层（Tool + Skill + System Prompt）  
约束 LLM 调用顺序、结果回报格式、错误传播与边界行为。

### 2.2 两条素材通道
1. 素材包通道（v1.0）：`asset_package_id + @文件名`
2. 媒体素材通道（v1.1A）：`%相对路径`

---

## 3. 部分一：界面改造（上传与引用体验）

### 3.1 入口改造
聊天输入框 `+` 菜单新增：

1. `Upload Assets`：上传单个或多个媒体文件（图/音/视频）。
2. `Upload Asset Folder`：上传文件夹并保留相对路径。
3. `Reference Assets`：查看可引用素材并插入到输入框。

### 3.2 交互行为
1. 上传素材成功后，可自动把引用写入输入框。
2. 引用格式统一为 `%相对路径/文件名`。
3. `Reference Assets` 支持目录导航（进入目录/返回上级）。
4. 非支持媒体类型在上传时直接拦截并提示。

### 3.3 与原始上传文件入口的关系
1. `Upload Files` 仍保留，用于通用文件上传。
2. `Upload Assets` 专用于视频生成素材链路。
3. 媒体文件在通用上传中默认不再走内容处理（`process=false`），避免无效错误日志。

---

## 4. 部分二：媒体处理流程（解析、存储、生成、查询）

### 4.1 主流程
1. 用户上传素材（单文件/多文件/文件夹）。
2. 后端将素材转为媒体资产记录并写入 TOS。
3. LLM 在 prompt 中使用 `%素材路径` 或 `@文件名`。
4. 先执行引用解析（resolve）：
   - 返回 `missing_references`
   - 返回 `ambiguous_references`
   - 返回 `available_references`
5. 校验通过后提交生成任务（默认异步，不阻塞等待）。
6. 通过任务列表或任务详情查询生成状态与结果地址。

### 4.2 媒体资产 API（v1.1A）
1. `POST /api/v1/media-assets/upload`
2. `POST /api/v1/media-assets/from-upload`
3. `GET /api/v1/media-assets/`
4. `GET /api/v1/media-assets/{asset_id}`
5. `GET /api/v1/media-assets/{asset_id}/url`

### 4.3 任务管理 API（素材包体系内）
1. `GET /api/v1/material-packages/tasks`
2. `GET /api/v1/material-packages/tasks/{task_id}`

说明：任务列表与任务详情支持状态刷新（特别是非终态任务）。

### 4.4 存储策略
1. 素材走 TOS。
2. 媒体素材与素材包分目录管理（避免路径混用）。
3. 媒体引用优先使用 `relative_path`，避免重名冲突。

### 4.5 关键防错策略
1. 缺失引用或重名冲突时，立即停止生成并回传可修复信息。
2. 任务提交后默认返回 `task_id`，由用户主动查询进度，支持并发创建任务。
3. 视频地址仅允许使用工具真实返回值，不允许推测或补全。

## 5. 编排配置（Tool / Skill / System Prompt）

### 5.1 Tool
统一维护：

1. `templates/seedance_material_package_tool.py`
2. `templates/seedance_material_package_tool_v2.import.json`

职责：
1. 素材包与媒体素材双模式工具调用。
2. resolve -> generate -> task query 的标准化输出。
3. 错误结构化回传（`status_code/error_code/error_message/request_id`）。

### 5.2 Skill
1. `templates/skills/seedance-execution-skill/SKILL.md`

职责：
1. 约束“先校验后生成”。
2. 约束“默认异步提交、手动查任务”。
3. 约束输出模板（表格、时间、字段完整性）。

### 5.3 System Prompt
1. `templates/prompts/seedance_system_prompt.txt`

职责：
1. 定义编排助手身份和边界。
2. 定义使用说明模式与执行模式路由。
3. 明确禁止脑补、禁止捏造结果。

### 5.4 编排文案共用经验（重要）
适用范围：Skill 与 System Prompt 的长期维护。

核心原则：只描述“当前最终生效行为”，不写开发历史。

执行规范：
1. 不写“已废弃/不再使用/曾经如何”的历史过程语句。
2. 只写“现在允许什么、禁止什么、默认怎么做”。
3. 输入约束用正向表达优先（例如“仅支持 `%素材路径` 引用”）。
4. 输出约束必须可验证（字段、格式、错误透传、是否允许补全）。
5. 任何未由工具返回的内容统一写 `暂无`，避免因历史上下文诱发模型补写。

这样做的目标：
1. 降低模型被历史语义干扰的概率。
2. 减少多版本迭代后的前后冲突。
3. 让提示词行为更稳定、更可回归测试。

## 7. 当前版本状态（v1.1A）

### 7.1 已完成
1. 界面入口改造（上传素材、上传文件夹、引用素材）。
2. 媒体资产独立 API 与 TOS 流程。
3. `%素材路径` 引用解析与冲突检测。
4. 任务异步提交与状态查询链路。
5. 任务状态刷新逻辑接入列表与详情。
6. 媒体上传默认禁用通用内容处理，减少无效报错。
7. 防幻觉约束：视频地址仅可使用工具真实返回值。

### 7.2 当前边界
1. 素材治理（批量清理、生命周期策略）仍可继续增强。
2. Tool/Skill/System Prompt 的生效依赖 OpenWebUI 控制台导入/更新，不会因文件改动自动热更新。

---

## 8. 运维与联调建议
1. 开发和生产严格分端口、分数据目录运行。
2. 每次后端改动后先跑路由检查再联调模型。
3. 每次更新 Tool/Skill/System Prompt 后，在 UI 内重新导入并做一次最小回归（上传、解析、提交、查任务）。
4. 对“已完成但无视频地址”的任务，输出必须是 `暂无`，并建议用户继续查询任务详情，不可生成占位链接。
