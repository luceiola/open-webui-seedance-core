# Seedance v1.1.3 需求文档（统一任务中心）

发布日期：2026-05-07  
修订日期：2026-05-08

## 1. 版本定位

1. `v1.1.3` 定位为“统一任务平台底座”版本。
2. 目标是让任务面板不再绑定单一模型（当前为 Seedance），并可纳入 HappyHorse 与后续任意 skill/tool。
3. 版本顺序：`v1.1.2`（任务中心首版） -> `v1.1.3`（统一任务平台） -> `v1.2`（素材库能力升级）。

## 2. 背景与问题

`v1.1.2` 任务面板已经可用，但仍存在结构性问题：

1. 任务数据源与 material-packages 路径耦合，不能天然覆盖 HappyHorse 与未来新增工具。
2. 不同 provider 的状态与返回格式不一致，前端难以稳定扩展。
3. 上游临时 URL 过期问题虽已缓解，但缺少统一任务模型与统一归档治理。

## 3. 目标

1. 建立统一任务模型，覆盖 Seedance、HappyHorse、未来所有 skill/tool 任务。
2. 建立 provider adapter 机制，把差异收敛在后端。
3. 统一任务状态、权限、下载路径与归档策略。
4. 前端任务页只依赖统一 `/api/v1/tasks/*` 契约。

## 4. 范围

### 4.1 In Scope（v1.1.3）

1. 统一任务存储（主表 + 产物表）。
2. 统一任务 API（列表、筛选、详情、预览、下载、删除、重试）。
3. Seedance adapter 改造为统一写入链路。
4. HappyHorse adapter 接入到统一写入链路。
5. 自动归档保持开启，平台下载统一从归档文件读取。
6. 旧任务数据回填脚本（`material_packages/*/tasks/*.json` -> 新任务存储）。

### 4.2 Out of Scope（v1.1.3 不做）

1. v1.2 素材库目录 ACL 重构。
2. 批量编辑、批量下载压缩包。
3. 复杂编排面板（DAG 可视化、高级调度策略）。
4. 多租户隔离策略重构（维持当前“全员可见”产品规则）。

## 5. 核心方案

### 5.1 统一任务主模型（建议字段）

1. 主键与来源：
   - `id`
   - `provider`（ark/happyhorse/xxx）
   - `provider_task_id`
   - `tool_name`
   - `skill_name`
2. 用户与上下文：
   - `user_id`
   - `user_name`
   - `chat_id`
3. 任务元信息：
   - `model`
   - `status`
   - `progress`
   - `error_code`
   - `error_message`
4. 治理字段：
   - `deleted_at`
   - `deleted_by`
   - `created_at`
   - `updated_at`
   - `finished_at`

### 5.2 统一产物模型（建议字段）

1. `task_id`
2. `artifact_type`（video/thumbnail/image/json/text）
3. `storage_backend`（local/tos）
4. `storage_path`
5. `mime_type`
6. `bytes`
7. `width`
8. `height`
9. `duration_ms`
10. `preview_ready`
11. `download_ready`

### 5.3 统一状态机

1. 任务状态：
   - `PENDING`
   - `RUNNING`
   - `SUCCEEDED`
   - `FAILED`
   - `CANCELED`
2. 归档状态：
   - `NOT_REQUIRED`
   - `PENDING`
   - `RUNNING`
   - `SUCCEEDED`
   - `FAILED`
3. 规则：
   - 任务 `SUCCEEDED` 后自动触发归档。
   - 任务下载只允许走平台归档文件流，不回传上游临时 URL。

## 6. 接口契约（v1.1.3 冻结）

字段级接口契约与错误码冻结见：

- `docs/seedance/32-v1.1.3-接口契约.md`

1. `GET /api/v1/tasks`
2. `GET /api/v1/tasks/providers`
3. `GET /api/v1/tasks/users`
4. `GET /api/v1/tasks/{task_id}`
5. `GET /api/v1/tasks/{task_id}/preview`
6. `GET /api/v1/tasks/{task_id}/download`
7. `DELETE /api/v1/tasks/{task_id}`（软删除）
8. `POST /api/v1/tasks/{task_id}/archive/retry`
9. `POST /api/v1/tasks/{task_id}/cancel`（建议实现）

`GET /api/v1/tasks` 查询参数（最小集）：

1. `user_id`
2. `provider`
3. `skill_name`
4. `tool_name`
5. `status`
6. `model`
7. `include_deleted`
8. `offset`
9. `limit`

返回格式冻结为：

```json
{
  "items": [],
  "total": 0,
  "offset": 0,
  "limit": 48
}
```

## 7. 权限规则（沿用当前产品口径）

1. 列表/详情/预览/下载：登录用户全员可见。
2. 删除/重试归档/取消：仅创建者或管理员。
3. 删除策略：软删除，保留期默认 7 天，超期物理清理任务与归档文件。

## 8. 兼容与迁移策略

1. 新增统一任务存储，不立即下线旧路径。
2. Seedance 先双写（旧 JSON + 新任务存储）降低风险。
3. HappyHorse 首先接入新任务存储，验证同面板展示。
4. 完成历史数据回填后，前端切到新统一接口。
5. 观察稳定后再下线旧任务读取路径。

## 9. 验收标准（DoD）

1. 同一任务面板可同时展示 Seedance 与 HappyHorse 任务。
2. 支持按用户筛选，且权限规则正确生效。
3. 任务成功后自动归档，历史下载不受上游 URL 过期影响。
4. 新增第三个 provider 时仅新增 adapter，不改任务前端页面。
5. 回填后旧任务可在统一列表查询与下载（已归档项）。

## 10. 切流门槛与回滚触发（冻结）

### 10.1 切流到新任务存储的准入门槛

1. 双写稳定运行至少 24 小时。
2. 双写失败率 < 0.1%。
3. 新旧任务记录抽样字段一致率 >= 99.9%。
4. 历史回填覆盖率 >= 99.9%（按任务数）。
5. `GET /api/v1/tasks` P95 延迟 <= 500ms（limit=48）。
6. 任务下载成功率 >= 99.0%（已归档任务）。

### 10.2 必须回滚到旧读取路径的触发条件

1. 新接口 5xx 比例连续 10 分钟 > 1.0%。
2. 双写失败率连续 10 分钟 > 1.0%。
3. 下载成功率连续 10 分钟 < 97.0%。
4. 发现权限越权（任一高危 case）且无法在 30 分钟内修复。

## 11. 存储治理阈值（冻结）

建议默认值：

1. `TASK_SOFT_DELETE_RETENTION_DAYS=7`
2. `TASK_ARCHIVE_RETENTION_DAYS=90`
3. `TASK_ARCHIVE_MAX_RETRIES=3`
4. `TASK_ARCHIVE_MAX_FILE_MB=2048`
5. `TASK_ARCHIVE_USER_QUOTA_GB=200`

容量水位告警：

1. `70%`：warning
2. `85%`：critical
3. `90%`：保护模式（新归档任务延后/限流，保下载优先）
